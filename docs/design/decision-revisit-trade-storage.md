## 1. 旧问题:当前是「点对点、卖家指定买家」的两阶段交易

[trade.proto](../../proto/pandora/trade/v1/trade.proto) `CreateOrder` 现在要求
`seller_id` + **`buyer_id`(指定买家)** + items + price,状态机:

```
PENDING ──买方确认──▶ BUYER_CONFIRMED ──卖方确认+结算──▶ COMPLETED
```

它的本质是**两个已知玩家之间的当面交易**:卖家挂单时就钦定了买家,双方各确认一次才成交。

| 维度 | 当前点对点交易 | 全服拍卖行 |
|---|---|---|
| 买家是谁 | 挂单时**已知**(`buyer_id` 指定) | **未知**,全服任意玩家竞买(跨人撮合) |
| 撮合 | 无,双方点对点 | **有**:挂单进公共市场,买家检索 + 抢购 |
| 并发模型 | 一单两人,几乎无竞争 | **一物多买家抢**,高并发争用同一挂单 |
| 数据规模 | 在途订单少、短 TTL | **常驻挂单大表**(全服活跃挂单百万~千万级) |
| 一致性焦点 | 双方各确认 | **「同一挂单只能卖一次」+ 买卖双方资源原子转移** |

**结论**:全服拍卖行不是给 `CreateOrder` 加个字段就行——它引入了
①**跨人撮合**(买家不预先指定)②**常驻挂单大表**(检索/翻页/筛选)
③**一物多抢的高并发争用** 三件当前架构没有的事。这三件事撞上的,正是好友图那套
"跨人强一致 vs 分片"矛盾,**外加一个买卖双方资源扣减的强一致 + 幂等要求**。

## 2. 新需求拆成三块

1. **挂单(List)**:卖家把物品挂到全服市场。物品必须**托管(escrow)**——挂单瞬间从卖家背包扣到
   "托管中",否则卖家可"挂单后把东西用掉 / 重复挂"作弊。
2. **撮合 / 抢购(Purchase)**:任意买家检索市场 → 对某挂单发起购买。同一挂单可能被**多个买家同时抢**,
   必须保证**只有一个买家赢**(挂单 `ACTIVE → LOCKED`),其余拿到"已被买走"。
3. **结算(Settle)**:赢家扣金币、卖家收金币、物品从托管转给买家。这是**跨两个玩家**的资源转移,
   必须**原子 + 幂等 + 可补偿**(不变量 #7)。

> 幂等贯穿三块:挂单可重试不重复挂、抢购可重试不重复扣金币、结算可重试不重复转移。详见 §5。

## 3. 核心矛盾:跨人强一致 + 高并发抢购 + 幂等,撞上分片

和好友图 [friend-distributed-scaling.md](./friend-distributed-scaling.md) §3/§4 同一个本质:
**结算要跨两个玩家(买家扣金币 / 卖家收金币 / 物品转移)做强一致写**。一旦按 `player_id` 分片:

- 买家货币、卖家货币、物品归属落在**不同分片 / 不同 slot**;
- Redis Cluster 跨 slot 无原子 `MULTI/EXEC` / Lua(报 `CROSSSLOT`,见 crossslot 文档 §3);
- 分库分表 MySQL 跨实例无本地事务,硬上 2PC/XA 在抢购热路径上不可接受。

但拍卖行比好友图**多一层难点**:**"同一挂单只能卖一次"是钱货交易,错一次就是刷钱 / 吞物**,
比"好友边最终一致"严格得多。所以**挂单状态机(谁抢到)必须强一致,资源结算必须幂等可补偿**,
不能像好友边那样简单"最终一致 + 软上限"了事。

## 4. 推荐形态:挂单单实体 CAS(强一致)+ 资源结算事务出箱(幂等最终一致)

把"谁抢到"和"钱货转移"解耦,各取所需:**抢购在挂单单实体上强一致 CAS,结算走幂等出箱**。

```
┌─ 步骤 1:挂单(List)——卖家分片本地事务,物品托管 ──────────────────┐
│  同一事务内:卖家背包扣物品 → 写"托管中" → INSERT listing(ACTIVE)   │
│  幂等键 = list_idempotency_key(client 生成 uuid):重挂同 key 直接返   │
│  原 listing_id,绝不重复扣物 / 重复挂(防客户端重试 / 双击)          │
└────────────────────────────────────────────────────────────────────────┘
            │ 挂单进全服市场大表(可检索/翻页/筛选)
            ▼
┌─ 步骤 2:抢购(Purchase)——挂单单实体 CAS,只有一个买家赢 ──────────┐
│  CAS:UPDATE listing SET state=LOCKED, buyer_id=?, lock_key=?         │
│       WHERE listing_id=? AND state=ACTIVE   (影响行数=1 才算抢到)     │
│  谁 CAS 成功谁锁定该挂单 → 天然防超卖 / 防并发双买                    │
│  幂等键 = buy_idempotency_key:同买家重试同 key,已 LOCKED 给自己 →    │
│           返回"你已抢到"而非"已售罄",重试安全                        │
│  CAS 成功的同一事务里写 outbox(settle 事件),保证"抢到 ⇒ 必结算"     │
└────────────────────────────────────────────────────────────────────────┘
            │ outbox → Kafka(key = listing_id / order_id,同单有序 §9)
            ▼
┌─ 步骤 3:结算(Settle)——跨人资源转移,幂等消费 ───────────────────────┐
│  consumer:买家分片扣金币 → 卖家分片收金币 → 托管物品转买家背包         │
│  幂等键 = order_id(= listing_id 派生):ledger 每步 INSERT IGNORE /     │
│           SETNX 防重复扣 / 重复发(不变量 #7);重试只生效一次          │
│  全部成功 → listing LOCKED→COMPLETED;余额不足 → 补偿:解锁挂单回      │
│           ACTIVE / 退还托管,发补偿事件,买家拿"余额不足"             │
└────────────────────────────────────────────────────────────────────────┘
```

**为什么这样切**:
- **挂单 / 抢购在单实体(listing 这一行 / 这一个 key)上做**,单分片合法、强一致、防超卖——
  和好友 §5 的 "request 单点 CAS" 同构。
- **结算是跨人写**,无法在抢购同一事务里跨分片原子完成 → 拆成 Kafka 事件 + 幂等消费,
  对齐 trade 现状已有的 `pandora.trade.audit` 出箱思路 + battle_result 的事务出箱(W4 ⑨)。

## 5. 幂等设计(用户明确要的,逐层落地)

幂等不是某一处加个键,而是**四条链路各有幂等键、各自只生效一次**。统一原则:
**每个会改状态的写都有幂等键,重复执行返回与首次相同的结果,不产生第二次副作用。**

| 链路 | 幂等键 | 落地手段 | 重试语义 |
|---|---|---|---|
| **挂单 List** | `list_idempotency_key`(client uuid) | 唯一索引 `uk(seller_id, list_idempotency_key)`;命中→读回原 `listing_id` | 重挂同 key 返回同一挂单,不重复扣物 |
| **抢购 Purchase** | `buy_idempotency_key`(client uuid) | CAS `WHERE state=ACTIVE`;已 LOCKED 且 `lock_key` 是自己→返回"已抢到" | 同买家重试不二次抢、不误报售罄 |
| **结算 Settle** | `order_id`(= listing 派生) | ledger 每笔 `uk(player_id, order_id, leg)` `INSERT IGNORE`;命中=已结算 | 重复消费只转移一次(不变量 #7) |
| **补偿 / 撤单 / 退款** | `order_id` + `leg=refund` | 同上,退款单独 leg 幂等键 | 补偿可重投,退款只退一次 |

### 5.1 三种幂等机制各管一段

1. **唯一索引兜底(挂单 / 结算腿)**:`INSERT ... INSERT IGNORE` / `uk` 冲突即"已做过",读回结果。
   这是最强的幂等——并发重复也只成功一行。
2. **条件 CAS(抢购)**:`UPDATE ... WHERE state=ACTIVE` 影响行数判赢,天然幂等 + 防并发超卖;
   重试时挂单已 LOCKED,用 `lock_key` 区分"是我抢的(返成功)"还是"别人抢的(返售罄)"。
3. **出箱保序(结算事件)**:CAS 成功与写 outbox 同事务(挂单分片本地),
   Kafka key = `listing_id`/`order_id` 保证同一单事件有序(§9),消费侧 ledger 幂等键再兜一层。

### 5.2 幂等键的生命周期

- client 生成的 `list_/buy_idempotency_key`:**服务端不可信不可猜**,落唯一索引;TTL 至少覆盖
  客户端最大重试窗口(建议 ≥ 24h,或与挂单 TTL 对齐)。
- 服务端派生的 `order_id`(snowflake):**结算与补偿全程同一个 order_id**,
  贯穿 ledger 各腿与 audit,是跨服务对账的主键。
- **铁律**:任何"扣钱 / 扣物 / 发货 / 退款"动作都必须挂幂等键落库,**没有幂等键的资源写一律不允许**
  (对齐不变量 #7)。

### 5.3 幂等的边界:强一致 vs 最终一致

- **挂单、抢购**:强一致(单实体 CAS,立即一致,不允许超卖)。
- **结算、补偿**:**at-least-once + 幂等 = 有效 exactly-once 效果**(秒级最终一致)。
  Kafka 短暂不可用时 outbox 行保留,恢复后重投,幂等键保证不重复转移——
  和 W4 ⑧/⑨ 的两段补偿链同一套可靠性模型。

## 6. 存储选型:沿用 friend 文档的结论

拍卖行的"跨人强一致结算"与好友图同源,选型结论也一致(详见 [friend-distributed-scaling.md](./friend-distributed-scaling.md) §6/§8):

| 用途 | 建议 |
|---|---|
| 挂单**权威大表**(百万~千万活跃挂单,需检索/筛选/翻页) | **关系库**(单 MySQL → 阶段 2 上 **TiDB** 几乎不改代码 → 阶段 3 分片)。Redis 不适合做权威主存(容量/持久性/跨 slot)。 |
| 市场**检索 / 列表缓存** | **Redis / 检索引擎** 缓存热门挂单页,主存仍是关系库。 |
| 抢购 CAS | 挂单按 `listing_id` 落单分片,CAS 单分片合法(MySQL 单行 / Redis 单 key Lua 皆可)。 |
| 玩家货币 / 背包 / 托管 | 按 `player_id` 分片;结算跨分片走 §4 步骤 3 幂等出箱,**不跨分片硬事务**。 |
| 结算事件 outbox | 与抢购 CAS 同分片写出箱表,key = `listing_id`/`order_id`(§9 同单有序)。 |

**阶段建议**(同 friend §8.4):
```
阶段 1(现在)        点对点两阶段交易,单 Redis      ← 当前位置,够用
阶段 2(拍卖行上线)   挂单大表上 MySQL/TiDB           ← TiDB 保挂单大表强一致 + 跨人结算用 §4 出箱
阶段 3(全服千万级)   挂单分片 + 结算异步幂等          ← 卸掉跨节点事务压力
```

> **TiDB 的价值**(friend §8):挂单大表的检索 + "同一挂单只卖一次"的强一致,TiDB 单 SQL 入口
> 即可保证,代码几乎不改;真正跨人的"钱货转移"仍走 §4 幂等出箱,不依赖分布式事务硬扛热路径。

## 7. 防作弊:物品必须托管(escrow),不是"挂个引用"

拍卖行最常见的刷物漏洞:卖家挂单后,在被买走前把同一物品**用掉 / 再挂 / 转走**,导致一物多卖 / 凭空复制。

- **挂单瞬间托管**:步骤 1 同事务内从卖家背包**扣到托管区**(`escrow`),挂单引用的是托管实例,
  卖家背包里已没有它 → 无法二次使用。
- **撤单 / 流拍**:挂单 `ACTIVE` 且未被抢 → 卖家可撤,**托管物品幂等退回**(`leg=cancel_refund`,order_id 幂等)。
- **结算失败补偿**:买家余额不足等 → 解锁挂单回 `ACTIVE` 或退还托管,全程幂等键防重复退。

> 这与不变量 #7"交易资源扣减必须原子 + 有补偿幂等键"完全一致;托管让"扣减"在挂单时就发生,
> 把作弊窗口关死。

## 9. 迁移成本(拍板后,预估分批实施)

这是一个**新功能域**,不是 `trade` 微调。预估改动面较大(proto + 新表 + 撮合 biz + 结算出箱 + 检索),
AGENTS §4 大范围改动虽不设文件数硬上限,但此新功能域跨 proto + 表 + 多服务且含不可逆上线风险,**仍建议分批**,不在本提案一次实现。粗分批:

1. **Batch A — proto + 挂单大表 + 挂单/撤单**:`AuctionService`(ListItem/CancelListing/SearchMarket/GetListing),
   挂单托管事务,客户端可见 `Listing` vs `ListingStorageRecord`,幂等键唯一索引。
2. **Batch B — 抢购 CAS + 结算出箱**:Purchase CAS + outbox + 幂等 ledger + 补偿,接 inventory / player 货币。
3. **Batch C — 检索 / 缓存**:市场分页 / 筛选 / 热门页缓存(Redis),价格区间 / 物品类目索引。
4. **Batch D(可选)— 竞价 / 一口价 / 手续费 / 税**:商业化规则。

> 现有点对点 `TradeService` **保留不动**(当面交易仍有用),拍卖行是**新增 `AuctionService`**,
> 二者并存,互不影响现有测试。

## 10. 验收标准(每批各自验,拍板时锁定)

1. **不超卖**:同一挂单被 N 个买家并发抢,**恰好一个 COMPLETED**,其余拿"已售罄",物品只交付一次。
2. **挂单幂等**:同 `list_idempotency_key` 重试,只产生一个挂单、只托管一次物品。
3. **抢购幂等**:赢家用同 `buy_idempotency_key` 重试,返回"已抢到"而非二次扣金币 / 误报售罄。
4. **结算幂等**:结算事件重复投递(模拟 Kafka 重放),买家金币只扣一次、卖家只收一次、物品只到一次。
5. **补偿幂等**:结算失败 / 撤单,托管物品幂等退回,重复补偿不二次退。
6. **托管防作弊**:挂单后卖家无法再使用 / 再挂同一物品。
7. **(上 TiDB / 分片后人工验)**挂单大表强一致 + 跨人结算最终一致秒级收敛,无单边账(扣了钱没发货 / 发了货没扣钱)。
8. `go build` / `go test ./...` 全绿;新增 biz 单测覆盖上述 1-6(fake ledger / fake repo,无需真 DB)。
