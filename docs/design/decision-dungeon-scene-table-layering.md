# 决策:副本/场景配表结构——分层方案(场景表 + 副本表引用场景 ID)

> 决策级别:配表规范(跨 策划 / UE DS / Go 服务端)。
> 触发:讨论「场景是副本就在副本表里填一个一样的 id、不用外键」这种旧项目做法是否可取。
> 日期:2026-07-06。状态:**已拍板——按分层方案执行**。
> 关联:`docs/design/` 副本选择链路见 Pandora-Client-SVN/Doc/服务器/副本选择_UE侧交接_Codex.md;
> 当前唯一关卡表为 Pandora-Client-SVN/Table/关卡/g_关卡.xlsx。

## 1. 背景与问题

旧项目做法:场景表与副本表**两张表、靠 ID 相等隐式关联**(某场景是副本,就在副本表里填一行
一模一样的 id),没有引用列、没有校验。讨论三个问题:这种做法有没有坑?对策划友不友好?
大厂怎么做?

先澄清:Excel 配表本来就没有真「外键」。所谓外键做法也只是副本表加一列 `场景id`,
靠**导表工具校验**。所以差别不在有没有外键,而在关系模型:

| | 同 ID 隐式对齐(旧项目) | 独立 ID + 场景id 引用列 |
|---|---|---|
| 关系 | 场景:副本 = 1:0..1(表结构层面写死) | 场景:副本 = 1:N |
| 策划心智 | 只记一个数字,最省事 | 两套 ID,稍多一步 |
| 场景复用 | ❌ 做不到 | ✅ 天然支持 |

## 2. 同 ID 隐式对齐的坑

它悄悄锁死一个假设——**一个场景只能是一个副本**。这个假设在游戏项目里几乎必然被打破:

- 同一张地图出普通/精英/噩梦三档副本(同场景、不同规则/掉落/怪物强度);
- 活动副本临时复用已有场景;
- 同场景 PVP 模式与 PVE 副本并存。

到那天只有两条烂路:复制场景表行凑新 ID(场景表膨胀、资源路径重复、改地图要改 N 行),
或当场重构成独立 ID(所有引用副本 ID 的地方全动)。且它把正确性寄托在
「ID 一样所以不会填错」的人为约定上,无任何校验兜底。

对策划:同 ID **初期反而更友好**(一个数字走天下),不友好发生在「想复用场景那天」。
独立 ID 的多填一列/填错风险,完全可由导表校验消掉(`场景id` 必须存在于场景表,填错不出包)。

## 3. 大厂主流做法(参照 WoW 数据表)

资源和玩法分两层,难度是第三个维度:

- **Map(场景表)**:一张地图一行,纯资源——umap/地形/寻路/光照。死亡矿井只有一行。
- **LFGDungeons(副本表)**:一个「可进入的玩法条目」一行,引用 `MapID`。
- **MapDifficulty(难度表)**:`MapID × Difficulty` 一行,配人数上限/CD/掉落加成。

普通/英雄死亡矿井 = 同一 MapID + 两条难度记录。国内大厂 MMO 配表结构大同小异,核心原则:
**资源表描述「它是什么」,玩法表描述「怎么玩它」,玩法表引用资源表**。
理由:美术做一张地图成本数周,策划开一档难度成本数分钟——成本差三个数量级的东西
不该挤在同一 ID 空间。

## 4. 备选方案对比(讨论过的三种)

### 方案 A:双表同 ID 隐式对齐(旧项目)
见 §2。唯一真正的坏方案——排序垫底。

### 方案 B:单表,每档难度 = 新地图 ID(旧项目实际演化形态 / 当前 g_关卡.xlsx 形态)
```
场景表: 1001 矿井·普通 → mine.umap
        1002 矿井·精英 → mine.umap   ← 同一资源路径,三行
        1003 矿井·噩梦 → mine.umap
```
- 能跑,对策划顺手(一个 ID 说清一切,跨表沟通/埋点/GM 命令只报一个数);
  很多二次元手游关卡表就是这种(每关一行、资源列重复)。
- 硬伤:资源信息重复(umap 路径/导航/出生点填三遍,美术改路径要改 N 行,漏一行=线上 bug);
  「场景表」名不副实,实际存的是「场景×玩法实例」,表大了找 umap 被谁用要肉眼筛。
- 补救:导表校验——**同一 umap 路径的多行,其余资源类字段必须一致,否则报错**。
  有这条校验,方案 B 完全可用。

### 方案 C:分层(场景表 + 副本表引用场景 ID + 难度维度)★ 拍板方案
- 「改地图」与「开难度」解耦,资源信息永远只有一份,不存在「改漏一行」的土壤;
- 规模再大也不腐化,WoW 二十年迭代验证;
- 代价:两套 ID,策划多一步——用导表引用校验兜底。

排序:**C ≈ B+校验 >> A**。前两名差距远小于它们与 A 的差距;A 的问题是把正确性寄托在人的自觉上。

## 5. 拍板结论

**按方案 C(分层)执行。**

配表结构:
1. **场景表**:场景 ID → umap 路径、寻路、出生点等纯资源字段。一张 umap 一行。
2. **副本表**:副本 ID(独立编号)→ `场景id`(引用列) + 玩法规则(game_mode、人数、
   难度、掉落、时限……)。同场景多档难度 = 副本表多行,引用同一场景 ID。
3. **导表校验(必须)**:副本表 `场景id` 必须存在于场景表,填错直接报错不出包。
   数据正确性靠导表校验兜底,永不靠「ID 恰好一样」的约定。

## 6. 对现有链路的影响(关键:服务端零改动)

- **匹配/DS 链路上透传的数字 = 副本 ID**(它决定规则与玩法),即现有全链路的 `map_id`
  语义收敛为「副本表主键」。Go 侧(StartMatchRequest.map_id → 撮合池按 map_id 分组 →
  ds_allocator → Agones label / PANDORA_MAP_ID env)**一行不改**——服务端从来只转发数字,
  不知道 umap 路径(既定铁律)。
- **DS(UE)读表逻辑**:由「map_id → 一行查出 umap」变为「副本 ID → 副本表行 →
  场景id → 场景表行 → umap + 规则」,两跳查表,仍全部落在表里。
- **客户端**:选副本 UI 列副本表条目,StartMatchRequest.map_id 填副本 ID。不变。
- g_关卡.xlsx 现状是方案 B 单表形态;落地时拆为场景表 + 副本表两个 sheet(或两个文件),
  由策划/UE 侧执行,导表校验随导表工具落地。
