## 1. 必修项（压测阻塞级）

### 【必修-1】登录关键链在 5s deadline 内串行扇出，任一下游变慢即大面积超时"进不去" · 登录/IO
- **位置**：`services/account/login/internal/biz/login.go:278`（`Login()`）、`login.go:657`（`resolveBattleAuthority` 无条件调 matchmaker）
- **已人工复核**：prod 请求 deadline = **5s**（`services/account/login/etc/login-prod.yaml.example:18`，dev 为 15s）；且每次**非战斗登录都会同步调一次** `matchmaker.ResolvePlayerMatchContext`（`login.go:657-660`）。
- **触发场景**：一次普通登录在**单个 5s ctx** 内串行完成：MySQL `FindByAccount`→`CheckBanned`→`PersistSessionJTI`（显式事务 + `SELECT ... FOR UPDATE`）→`TouchDevice`（4 次往返）＋ Redis `sessions.Set`/`GetJTI`（2 次）＋ **3 次跨服务 gRPC**（`player_locator.GetLocation`、`matchmaker.ResolvePlayerMatchContext`、`hub_allocator.AssignHub`）。全部串行、无子预算切分。基线本身已吃掉约 1~2s。压测 ramp/稳态时任一下游（尤以 matchmaker 最繁忙）P99 抖到几百 ms~数秒（GC/锁竞争/leader 切换），串行相加越过 5s → ctx 取消 → **登录整体失败，玩家进不去**。
- **为什么防线不生效**：客户端 15s 超时被请求已带的 5s deadline 盖住而失效；SRE 熔断只对**错误率**跳闸，对秒级**慢成功**不触发。
- **建议修法**（注意安全边界）：
  1. 给 `locator` + `matchmaker` 两个**只读探测**独立于 hub 分配的短 sub-deadline（如 2~3s），且必须在 data 层各调用点显式 `context.WithTimeout(ctx, budget)`——靠 dial option 的 `WithTimeout` 在继承 deadline 时是 no-op。
  2. ⚠️ **不要盲目"普通登录不查 matchmaker"**：该无条件查询是专门为封 codex P0-2/3/4 窗口（READY 与 notifyBattle 投影间窗口、locator 30s TTL 蒸发时对局仍活跃）加的；B1 enforce 档对 matchmaker 失败是**故意 fail-closed**。任何"读超时即 presence-only 降级"只能对 `local/off` 档生效，enforce 档放开会重开 P0 不安全路径。
  3. `TouchDevice`、`NotifyLoginPending` 等**非鉴权副作用**可评估改 detached background（`context.Background` 派生 + 白名单 trace/player 字段，遵 §16.7），移出鉴权关键路径。
- **验证**：CONFIRMED（C4）+ 人工复核 5s deadline 与 matchmaker 每登录必调。

### 【必修-2】ds_allocator 的 `ownerAdmitted` sync.Map 只增不删，压测数小时后 OOM · 内存
- **位置**：`services/battle/ds_allocator/internal/biz/owner_authority.go:119` 与 `:132`（`admitted.Store`）、`allocator.go:147`（定义）
- **已人工复核**：ds_allocator 全服务只有 `admitted.Store`，**没有任何** `admitted.Delete` / `admitted.Range` / `sweepStaleOwnerAdmitted`。而同构的 `hub_allocator` 对同名字段**有完整清扫**（`hub_allocator/internal/biz/owner_authority.go:40` `ownerAdmittedStaleTTL=5min`、`:45` `sweepStaleOwnerAdmitted`、`hub.go:1608` 周期调用），注释自称"复审 P1-5……防缓存随历史实例 UID 无界增长"。**ds_allocator 明显漏 port 了这段。**
- **门控**：`owner_addr` 已配置（`main.go:292` `SetOwnerAuthority`，即 §9.22 owner 权威方向启用）。未配则 `ownerAdmitCensusWeak` 在 `auth==nil` 提前返回，不泄漏。
- **触发场景**：每场 Battle 对局分配一个**全新 InstanceUID** 的 DS，授权心跳里玩家达 `ADMITTED` 即 `admitted.Store(InstanceUID|playerID, struct{}{})`。DS 打完销毁、**UID 永不复用**，条目却永久驻留。数十万场对局 × 每局玩家数 → 数百万僵尸条目 → 进程 RSS 单调上涨不回落 → OOM 被 k8s 杀 → 分配服务重启后一段时间新对局无法分配 Battle DS。
- **建议修法**：**直接照搬 hub_allocator**（低风险，同仓已验证代码）——`admitted` 值改存 `time.Time`（last-touch）+ 命中续期；新增 `sweepStaleOwnerAdmitted` 在心跳/sweep 循环周期 `Range` 删除 last-touch 早于 cutoff（5min）的项；同时补上 hub 版 `ownerAdmitCensusWeak` 里的"函数内按本实例剪枝 Range"（ds 版连这段也缺）。
- **验证**：CONFIRMED（C1）+ 人工复核 Store/Delete 缺口与 hub 参照实现。

### 【必修-3】matchmaker DS 分配在单 leader 协程内串行阻塞，压测下撮合吞吐坍塌 · 背压/IO
- **位置**：`services/matchmaking/matchmaker/internal/biz/match.go:2517`（`advanceAllocationsOnce` for 循环内联同步 `advanceAllocation`）、`match.go:1735/1742`（内联调 `AllocateBattle`）、`main.go:304`（`RunMatchLoop` 单协程）
- **触发场景**：同一 `(game_mode, region)` 分片的所有对局只由**一个 leader 协程**推进；分配 RPC per-call 超时默认 **60s**（agones allocate 5s + DS ready_wait 45s，`conf.go:157`）。压测爆发式开局把 agones warm buffer 耗尽 → 新对局等冷 pod ready（数秒~数十秒/局）→ 串行 for 循环把分配吞吐钳到"1 次/单次分配时延"（如 5~10s → 每分钟仅 6~12 局），并**头阻塞**同分片的新成局（`matchOnce`）、READY 补推、过期清扫 → 海量玩家卡在匹配/READY，分钟级停滞。
- **有界性**：60s per-call 有界、pool 预热后自愈、match TTL 30min 远大于延迟（玩家不被丢弃只是等），故非 P0；但负载尖峰下确实违反"玩家不卡"精神。
- **建议修法**：把 `advanceAllocationsOnce` 内的 DS 分配剥离到**有界并发 worker 池**（信号量 N≈16~64），主循环只做状态推进与派发，分配结果仍经 `UpdateMatchWithLock` CAS 回写 canonical match（`operation_id` 幂等语义不变，单写者约束由 CAS 守住）；给每个分配 goroutine 派生带 timeout+cancel 的 detached ctx（§16.7）。另建议给每 tick 分配总耗时设预算，让新成局不被分配批阻塞。
- **验证**：CONFIRMED（C3）。

### 【必修-4】撮合循环每 2s 全量拉取整个排队 ZSET + 逐票串行 GET，队列无准入上限 · 背压/CPU
- **位置**：`match.go:2549`（`RangeQueueTickets`）、`data/match.go:533`（`ZRANGE queue 0 -1` 无 LIMIT）、`biz/match.go:2559`（逐票 `GetTicket` 单条 GET）、`biz/match.go:2575`（`sort.SliceStable` 全量排序）、`biz/match.go:422`（`StartMatch` 入队无容量准入）
- **触发场景**：单人排队 / PVE（`team_size=1`）/ 奇偶或副本约束模式，票据配对受限而滞留；或活动开场集中登录突发入队。队列堆到约 1e4 时，单 tick = N 次串行 Redis GET（LAN 上 ~0.1~0.5ms × N ≈ 2s）+ N 次 proto Unmarshal + O(N log N) 排序 ≥ `match_interval`（2s）→ tick 堆积 → 撮合频率下降 → 若入队速率持续高于抽干速率则**正反馈雪崩**，同 tick 的 reconcile/advance/expire 一并被拖慢。**压测标准做法（阶梯 ramp 数万并发进单一模式队列）正落在此场景内。**
- **建议修法**：① `RangeQueueTickets` 加 `ZRANGE ... LIMIT` 或按 avg_mmr score 分段游标，单 tick 只处理有界数量并轮转起点（batch 上限须 ≥ 单 tick 稳态入队量，否则永远追不上）；② 票据加载按 slot 分批 pipeline/MGET（注意 Cluster 跨 slot：queueKey 与 ticketKey 不同 slot）；③ `StartMatch` 入口对每 `(game_mode, map_id, mmr桶)` 队列长度做准入上限，超限返回可重试忙错误（§9.18 精神）；④ 加单 tick 耗时/队列深度 metric + 告警。
- **验证**：CONFIRMED（C5）+ CPU 侧同源（P8）。

### 【必修-5】世界频道聊天无发送频率限制 + 单条消息 O(N) 广播扇出，可拖垮整个 push 层 · 背压
- **位置**：`services/social/chat/internal/biz/chat.go:96`（`SendMessage` 只校验长度，无 per-player 冷却）、`chat.go:333`（`sendWorld` 广播）、`services/runtime/push/internal/biz/connection.go:110`（`Broadcast` 每消息 `make([]*StreamSlot,0,len(bySlot))` + 逐 slot channel send）
- **触发场景**：任一持有效 session 的客户端（或改包脚本）高频刷 WORLD 频道，无任何拦截 → 每条消息经 `PushWorld`→kafka 广播 topic → **每个 push Pod（各自独立 consumer group）都 Broadcast 一次**，每次分配 `len(本Pod在线连接)` 的指针切片 + 全量非阻塞 channel send。集群级成本 ≈ 发送速率 × 全服在线数，叠加每消息切片分配的 GC 压力 → push 进程 CPU/GC 吃满 → **定向推送（READY/私聊/系统）一起变慢积压** → 退化面覆盖全服在线玩家，而非仅刷屏者。
- **注意**：每 Pod 独立 consumer group，**单 Pod 自限无效**，限速必须在 chat 生产侧或广播 topic 消费侧统一做。
- **建议修法**：① chat 源头对 WORLD 加 per-player Redis 令牌桶（治本，直接压掉扇出次数）+ 世界频道全局速率上限/合并窗口，超限返回 `ERR_RATE_LIMITED`；② push `Broadcast` 复用缓存 slot 快照，避免每消息重新分配切片。
- **验证**：CONFIRMED（C6）。

### 【必修-6】server-stream RPC 与所有后台 goroutine 无 panic recover，单点 panic 崩整进程 · panic
- **位置**：`services/runtime/push/internal/service/push.go:46-49,58`（Subscribe 是 server-stream，注释自证 unary 中间件链对它不生效）、`biz/push.go:410`（看门狗 `go func` 无 recover）、`match.go:2182`（`RunMatchLoop`）、`player_locator/.../presence.go:204`、`hub_allocator/.../hub.go:1592`（`RunHeartbeatSweep`）。全仓**唯一** recover 在 `pkg/middleware/recovery.go`，只覆盖 gRPC unary。
- **触发场景**：任一 latent panic（nil 解引用、index 越界、类型断言失败、迟到回调解引用）命中 push Subscribe、其派生看门狗、或 matchmaker/allocator/locator 的后台循环 → 因该路径无任何 recover → **整进程崩溃**（非单请求降级）。push 崩 = 该 Pod 数十万长连瞬断 → 客户端集中重连风暴雪崩；`RunMatchLoop` 崩 = 撮合彻底停摆，玩家匹配卡死。
- **重要修正**：**并发读写 map 触发的是 fatal runtime throw，`recover()` 本就兜不住**——那类必须靠正确同步，别指望 recover。recover 只能兜可恢复 panic（nil/越界/类型断言/close 已关 channel 等）。
- **建议修法**：① 为 server-stream 增加带 recover 的 `kgrpc.StreamMiddleware`（或 Subscribe 入口 `defer/recover` 转 gRPC 错误），让单 stream panic 只断该连接；② 提供统一 `SafeGo`（recover 后打 stack + 计数，按策略续跑或有界退出）包装**所有**长生命周期/后台 goroutine（看门狗、RunMatchLoop、PresenceHub tick、RunHeartbeatSweep、kafka 消费、fire-and-forget）。
- **验证**：PLAUSIBLE（P14）——后果链已被代码完全证实，但审计**未定位到一条"压测下确定被执行到"的具体 panic 语句**，故列 PLAUSIBLE。**一旦在热路径（`stream.Send`、`drainBuffer` 解析 Redis/kafka payload、游标运算、迟到回调解引用）定位到任一可复现 panic，严重度立即上调 P0。** 建议压测期专门盯 push/matchmaker 的崩溃重启计数。

## 2. 发布配置门禁（不是代码 bug，是上线/压测配置必须核对的项）

### 【门禁-A】服务端 BBR 自适应限流默认关闭 · IO/背压（P5）
- `pkg/grpcserver/grpcserver.go:55`：`RateLimit()` 被 `c.Grpc.EnableRateLimit` 门控，仓库内**无任何 yaml 置 true**（只有 `*-dev.yaml`）。过载时无 server 侧丢负载。
- 后果被三层机制钳住（爆炸半径有界、可自愈，**非宿主级静默雪崩**）：每 RPC 15s 有界超时 + Envoy 默认 ~1024 并发隐式熔断 + k8s 256Mi 内存硬限触发 OOMKill→重启。真实表现是个别 pod 延迟劣化/可能 crash-loop，不是全集群崩。
- **必做**：上线/压测配置对**所有对外可打流量的 gRPC 服务** `enable_rate_limit: true`（BBR 无需配阈值）；压测预案断言 `pandora_rpc_total{code=RATELIMIT}` 与 RT 收敛。加入发布 checklist（与 `enable_reflection=false`、TiDB 门同级）。

### 【门禁-B】MySQL 连接池默认偏小，登录风暴下池饱和排队 · 登录/IO（P3+P11）
- `pkg/mysqlx/mysqlx.go:30`：默认 `MaxOpenConns=32`/`MaxIdleConns=8`；`login-prod.yaml.example` 为 64。单登录 4 次 DB 往返，其中 `PersistSessionJTI`（`session_generation.go:76`）是**显式事务 + `SELECT ... FOR UPDATE`**，持连接 + 行锁时间最久。
- **触发**：数百级在途登录，或任一查询被拖慢（MySQL 锁等待/同账号并发登录在 FOR UPDATE 行锁上串行）→ 池打满 → `database/sql` 连接获取排队至请求 ctx（prod 5s）到期 → **成批登录超时失败，而 CPU/MySQL 本身未满，瓶颈在池配额**。
- **必做**：按目标登录 QPS × 真实持连时长核定 `max_open_conns`（与 MySQL `max_connections` 一起放大）；设 `SetConnMaxIdleTime`；把 `DBStats.WaitCount/WaitDuration` 作为压测断言项。**优化收益最大**：把 `PersistSessionJTI` 的 SELECT-FOR-UPDATE+回读评估为单条原子 UPSERT，减少事务期持连与行锁。

### 【门禁-C】Redis 池参数不可配 + 关键服务未配 read/write_timeout + ctx deadline 不约束 socket IO · 登录/IO（P4+P6）
- `pkg/redisx/client.go:94/105`：只设 Dial/Read/WriteTimeout，**未设** PoolSize/MinIdleConns/PoolTimeout/MaxRetries；`config.RedisConf` 根本没有这些字段（运维无法调优）。默认走 go-redis：PoolSize=`10×GOMAXPROCS`、PoolTimeout=ReadTimeout+1s。且 `ContextTimeoutEnabled=false`（仅 auction 的 `NewDeadlineUniversalClient` 打开）→ 业务 ctx deadline 不作用到单次 Redis socket IO。`login-dev.yaml` 只配了 `dial_timeout:2s`，read/write 走默认 3s。
- **触发**：Redis 热点 key/慢 Lua/failover → 命令占连接到 ReadTimeout（3s）才释放 → 并发命令超 `10×GOMAXPROCS` → 后续命令排队 → 约 4s（PoolTimeout）后 `connection pool timeout`。login 的 session/JTI 校验强依赖 Redis 且不可降级 → 登录短时失败/卡顿，且**无配置项可临时扩池自救**。
- **必做**：`RedisConf` 增加 `pool_size/min_idle_conns/pool_timeout/max_retries` 并在 redisx 透传；login/session/locator 等强依赖显式配 `read_timeout/write_timeout=1s`（与 matchmaker/battle 对齐，把最坏 socket 占用从 3s 降到 1s）；评估 session 读写路径走 `NewDeadlineUniversalClient`；**引入 `go.uber.org/automaxprocs`**（否则 k8s 限核下 GOMAXPROCS 取节点核数，池大小与实际可用 CPU 脱节）。
- 注意：go-redis 在 ctx 取消/超时时把连接判 badConn 并**关闭**（非复用），故收益是"更快 fail 并暴露"，不是"更快归还复用"。

## 3. 加固 / 次要项（P2，压测常规规模不必然触发，但值得修）

### 【P2-1】kafka 消费循环 Consume 出错无退避 + handler 无 recover · CPU/panic（P1/P7 + P13）
- `pkg/kafkax/consumer.go:163`：Consume 出错紧重试无 backoff。**实际被 sarama 内部节流**（metadata 3×250ms、rebalance 4×2s），表现约 1 行/秒错误日志、CPU 可忽略，**非满速自旋打满单核**（原审计头条被验证收窄）。真实缺口是：无 jitter → broker 恢复瞬间多服务同时重连轻度惊群；错误日志无聚合刷屏。
- `consumer.go:244`：`processMessage` 无 `defer/recover`。解码边界已用 `Poison→DLQ+ack` 防住畸形 payload（proto.Unmarshal 对任意字节只返 error 不 panic）；**残余风险 = 解码成功后深层业务的确定性 panic → 重放同 offset → CrashLoop**。
- **修法**：Consume 出错后 `select{ case <-ctx.Done(); case <-time.After(backoff) }`（指数退避封顶 ~30s + 成功重置）+ Errorf 首错/每 N 次限频 + Consume 快返 nil（空 assignment/反复 rebalance，**最会空转的正是这条不打日志的路径**）加极小 sleep 兜底；`processMessage` 加 `defer/recover` 打完整 stack + 按毒丸投 DLQ + ack。

### 【P2-2】bcrypt 登录热路径同步执行、无并发上限 · 登录/CPU（P10）
- `login.go:298` `passwd.Verify` 同步跑 bcrypt。**已人工复核**：全仓账号创建（`login.go:856` `ensureAccount`）用的是 `passwd.DevCost`（cost=4，单次 ~5ms），**`ProdCost=10` 常量零引用**——真实雪崩阈值比"cost=10（60~100ms）"高 ~15~20 倍。冷登/顶号/首登压测下仍是 CPU 密集，无 semaphore/BBR 卸载。
- **修法**：开 BBR（【门禁-A】，最小改动）+ 可选给 bcrypt 加并发 semaphore（上限≈核数×小系数），超限快返 `ErrUnavailable`。
- **⚠️ 独立安全问题**：`ProdCost=10` 从未被使用，若生产口令实际以 cost=4 落库，是**口令强度弱点**，应单独确认并修（属安全，不属压测）。**压测复现必须用生产真实 cost 的存储 hash，不能用 `dev_skip_password=true` 掩盖。**

### 【P2-3】登录内串行 sleep 重试放大在途 goroutine · 登录（P12）
- `login.go:605`：`queryBattleLocation` 对 locator 错误 3×50ms 重试；`hub_client.go` `AssignHub` 对 Unavailable 3×150ms 重试。均在登录 ctx 内串行 `time.After` 阻塞。
- 收窄后：**locator 腿**是真实但有界（常数 3×、单登录额外纯等待 ≤ ~400ms、双 `ctx.Done()` 钳住）的 retry storm 放大器（login 无 BBR/熔断过载联动）；**hub 腿**只对显式 Unavailable 重试，过载常见的慢响应走 fast-fail，基本不放大。不构成"自我加剧拖垮"的正反馈。
- **修法**：给 locator 腿重试接入过载/熔断信号，过载判定成立时跳过重试直接 fast-fail，确保重试总时长计入登录延迟预算。

### 【P2-4】snowflake 单节点 32768 ID/秒硬顶，耗尽后 1ms sleep 轮询 · CPU（P9）
- `pkg/snowflake/snowflake.go:49`：秒级时间戳 + 15bit step，单副本每秒 32768 ID 上限；耗尽后 `waitNextTime` 以 `for { time.Sleep(1ms) }` 轮询到下一秒（goroutine 短暂 park，不烧 CPU，自恢复）。
- 只有"单副本独吞热流量且同一秒需 >32768 ID"才触发；mail 批量附件 burst 最接近，但受附件数上限 + journal 写先成瓶颈兜住。**常规压测不触发。**
- **修法**：证实存在真实需求前**不改**（§15 拒绝预设复杂化）；加"进入 waitNextTime 的次数/耗时"监控作容量逼近信号。真需扩容再把时间戳改毫秒级。

### 【P2-5】push 跨 Pod 唤醒用单一 Redis pub/sub channel 全量广播 · 背压（P15）
- `services/runtime/push/internal/biz/consumer.go:274`：每条定向推送无条件 `PublishWake` 到固定 channel `pandora:push:wake`（`data/wake.go:30`）；所有 push Pod 订阅同一 channel → M 倍放大。
- 收窄：M 被"分 Cell 架构"钳到 6~8（>40 万 CCU 分 Cell，各 Cell 独立 Redis+push pod，不随总 CCU 无界增长）；群发走 Broadcast 不发 wake；稳态定向速率（0.05~0.2/s/玩家）下单 Redis 不饱和。后果是 best-effort 降级到 ≤30s 兜底轮询（帧恒在缓冲，不丢不脑裂）。`wake.go:14-16` 已按 §15 刻意推迟分片方案。
- **修法**：先加订阅 fan-out / `onWake` 落后 + go-redis Channel drop 计数的 Prometheus 观测（当前无，饱和会静默退化到轮询）；达量化门槛再上分片 channel。

### 【P2-6】player_locator presence 订阅倒排索引无断连清理/无 TTL · 内存（C2）
- `presence.go:85-86`：`watchers`/`watching` 唯一删除点是显式 `Unsubscribe` RPC（`presence.go:165`）；无 TTL、无断连回收、无周期 sweep；Subscribe/Unsubscribe 是 unary，不绑长连生命周期。玩家自身下线走 `ClearLocation→Notify(Offline)`，把 playerID 当"被关注者"推，**不清理它作为"订阅者"的条目**。
- **门控**：`presence.enabled` 默认关（当前生产不走此路径）+ 该 fan-out 明确单实例限定。启用后随累计（非并发）订阅玩家增长，滚动重启附带封顶。故 CONFIRMED 但 P2。
- **修法**：把订阅生命周期绑定到 push 长连接（连接断开自动 Unsubscribe，与本文件架构注释一致），或给每 subscriberID 记 last-seen + `Start()` tick 周期 sweep；同时对订阅者数量加单实例上限（§9.18）。

### 【P2-7】matchmaker reconcile 每 5s 对撮合 Redis 全键空间 SCAN · 背压/CPU（P16）
- `data/match.go:961`：`scanAllMasters` 对每个 master 全键空间 `SCAN MATCH`（`biz/match.go:2192,2201` 每 5s）。SCAN 成本随该 Redis 实例总键数线性增长。
- 收窄：撮合键是**有界工作集**（终态删键/FAILED 挂 TTL/claim TTL），数万键单趟亚秒、Redis CPU 影响轻微；只有在途工作集达十万级以上才可见退化。**压测常规规模不造成撮合退化。**
- **修法**：常态 reconcile 改用维护中的 due/active ZSET 索引驱动（已有 `RangeDueStartOperations`/`RangeActiveMatches`），全键 SCAN 降为分钟级漂移兜底；更该优化的是 `reconcileStartOperationsOnce` 对每个存活 op 无差别每 5s 重发逐成员 `ClaimStartPlayer`（仅在 due/claim 缺失时重建）。

## 4. 交叉印证的独立结论（人工复核，非工作流）

- **登录 prod deadline=5s / dev=15s**（`login-prod.yaml.example:18`）——【必修-1】的关键前提，已核实。
- **infra 组件（Redis/MySQL/etcd）在 `deploy/k8s/infra` 为单副本**——dev/压测环境无 HA，压测中依赖单点抖动会直接反映为上层退化，属预期，压测前确认容量。

## 5. 已核查并排除（现有机制已防住，无需改）

- **R1 拍卖 256 路条带锁串行 Redis+gRPC**（`auction.go:249`）：锁序一致、单锁持有、Redis 抢锁 3s / Settle 5s 超时 / reserve 冲突上限 64，均有界自愈，**非死锁**；健康 inventory 下持锁亚毫秒，256 条带足够。仅 inventory 结算显著变慢时碰撞 market 尾延迟放大（那才是主事故）。
- **R2 local_allocator 持 `l.mu` 期间 exec 拉起 DS**（`local_allocator.go:145`）：生产固定走 `mode=agones`，local 被门控在 `ds_auth.mode=off + authority_mode=legacy` 的单机自测 profile（`main.go:261`），**永不作为上线路径**；`cmd.Start()` 仅 fork+exec，`Wait()` 在锁外，token 是本地 JWT 无 Redis 往返；资源（RAM/port）才是墙。**非死锁。**
- **R3 kafka 内联阻塞重试造成分区 HOL**（`consumer.go:258`）：掉落/MMR/release 走后台出箱（有 §9.24 outbox 无堆积告警），battle_result 分区消费不受下游慢影响；内联 `GetMMR` 遇慢优雅回退 `BaseMMR`（每玩家至多一次 + 熔断）。残余（非 Poison 瞬时错误串行阻塞单分区）触发源是 MySQL brownout 非 gRPC 慢，≤3×500ms+DB 超时即进 DLQ 自排空，有界自愈。
